# Liquid Glass Motion Library App PRD + MVP Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** iOS 네이티브 Liquid Glass UX/UI를 기반으로, 모션 프리셋을 탐색/미리보기/저장/적용할 수 있는 Motion Library MVP를 빠르게 출시한다.

**Architecture:** Expo Dev Client + local Expo module(SwiftUI) 중심으로 구현한다. 핵심 화면/내비게이션/애니메이션은 SwiftUI 네이티브 컴포넌트로 구성하고, JS 레이어는 데이터 주입/실험 플래그/로깅에 집중한다.

**Tech Stack:** Expo 55, React Native 0.83, @expo/ui, SwiftUI, Expo Modules, iOS Simulator, local JSON(초기)

## 1) 앱 개발 내용 정리 (Product Summary)

### 1.1 제품 한 줄 정의
- "iOS Liquid Glass 경험을 갖춘 모션 레시피 라이브러리 앱"

### 1.2 사용자 문제
- 팀 내에서 재사용 가능한 모션 패턴이 문서/코드/디자인에 분산되어 있다.
- 디자이너/개발자가 동일 모션을 재현할 때 결과가 일관되지 않다.
- iOS 네이티브 감성과 어긋난 커스텀 애니메이션이 자주 발생한다.

### 1.3 MVP 사용자
- Primary: iOS 앱 디자이너/개발자(프로토타입/실서비스 UI 구현 담당)
- Secondary: PM/QA(상호작용 기준 합의, 회귀 체크)

### 1.4 핵심 가치
- 모션 패턴을 "검색 가능한 자산"으로 만든다.
- 미리보기와 적용 코드(파라미터 포함)를 한 화면에서 연결한다.
- iOS HIG/SwiftUI 시맨틱을 기준으로 기본 퀄리티를 보장한다.

## 2) PRD

## 2.1 문제 정의
현재 앱은 UI 복제 단계는 빠르게 진행되고 있으나, 모션 시스템이 제품 자산으로 구조화되지 않았다. MVP는 "탐색 → 미리보기 → 저장"의 가장 짧은 가치 루프를 완성해야 한다.

## 2.2 목표 (MVP)
- G1. 모션 프리셋 20개를 카테고리 기반으로 탐색 가능
- G2. 프리셋 상세에서 실시간 미리보기 + 파라미터 조정 가능
- G3. 즐겨찾기 저장 및 재조회 가능
- G4. iOS 네이티브 내비게이션/탭/시트 패턴 준수

## 2.3 비목표 (MVP 제외)
- 사용자 계정/클라우드 동기화
- 팀 협업 권한/코멘트
- 서버 기반 추천 알고리즘
- Android 동시 런칭

## 2.4 성공 지표
- Activation: 첫 실행 후 3분 내 프리셋 상세 진입률 >= 70%
- Engagement: 세션당 프리셋 미리보기 실행 횟수 >= 4회
- Retention proxy: 즐겨찾기 저장 비율 >= 30%
- Quality: 핵심 화면 크래시율 0%, 주요 UI 회귀 0건

## 2.5 핵심 사용자 시나리오
1. 사용자가 홈에서 카테고리 탭을 전환한다.
2. 카드 목록에서 모션 프리셋을 선택한다.
3. 상세 화면에서 예제 컴포넌트에 모션을 재생/파라미터 조정한다.
4. 마음에 드는 프리셋을 즐겨찾기에 저장한다.
5. 즐겨찾기 탭에서 저장 목록을 다시 확인한다.

## 2.6 기능 요구사항

### F1. Library Home
- 카테고리 세그먼트(예: Entrance, Emphasis, Exit, Gesture)
- 프리셋 카드 그리드(썸네일, 이름, 태그)
- 검색(이름/태그 기준)

### F2. Motion Detail
- 실시간 미리보기 영역
- 조정 가능한 파라미터(기간, 딜레이, 이징, 반복)
- 적용 코드 스니펫(읽기 전용, 복사 버튼은 v1.1)

### F3. Favorites
- 즐겨찾기 토글
- 즐겨찾기 목록 필터/정렬(최근 저장순)

### F4. Native UX 준수
- SwiftUI NavigationStack + TabView 기반
- Liquid Glass 스타일 계층 유지(탭/내비게이션 top layer)
- 접근성: 섹션 헤더 trait, 44x44 탭 타깃 준수

## 2.7 비기능 요구사항
- 성능: 상세 미리보기 인터랙션에서 프레임 드랍 체감 최소화
- 안정성: iOS dev build에서 재현 가능한 동작
- 확장성: 프리셋 데이터 모델을 JSON->Remote로 확장 가능하게 설계

## 2.8 정보 구조
- 탭: `라이브러리` / `즐겨찾기` / `설정`
- 라이브러리:
  - 카테고리 세그먼트
  - 프리셋 카드 목록
  - 프리셋 상세(푸시)

## 2.9 리스크와 대응
- R1. 네이티브/JS 브릿지 복잡도 증가
  - 대응: MVP는 네이티브 렌더 중심, 데이터만 JS 주입
- R2. 모션 과다로 UX 피로
  - 대응: 모션 강도 프리셋(Soft/Default/Expressive) 제공
- R3. 디자인-코드 불일치
  - 대응: HIG 체크리스트 기반 리뷰 게이트 추가

## 2.10 릴리즈 범위 (MVP)
- 포함: 카테고리 탐색, 상세 미리보기, 파라미터 조정, 즐겨찾기
- 제외: 로그인, 공유, 클라우드, 협업

## 3) 접근 방식 옵션 (2~3안)

### A안. SwiftUI Native-First (추천)
- 핵심 UI/모션을 SwiftUI에서 구현, Expo는 호스팅/데이터 전달
- 장점: iOS 네이티브 품질, Liquid Glass 대응 유리
- 단점: 초기 브릿지 설계 필요

### B안. Hybrid (JS 화면 + Native 미리보기 위젯)
- 목록/탐색은 RN, 상세 미리보기만 SwiftUI
- 장점: RN 생산성 + 핵심 네이티브 품질
- 단점: 화면 일관성/상태 동기화 복잡

### C안. RN-Only 커스텀 구현
- 모든 화면을 JS 레이어에서 구현
- 장점: 개발 속도(초기) 빠름
- 단점: Liquid Glass/HIG 네이티브 감성 재현 리스크 큼

**추천:** A안. 현재 코드베이스(Expo + local SwiftUI module)와 가장 자연스럽고 기술 리스크가 낮다.

## 4) MVP 빠른 개발 계획 (10 영업일)

### Phase 0 (Day 0.5): 범위 고정
- 프리셋 데이터 스키마 확정(id, category, tags, duration, easing, intensity)
- MVP 포함/제외 확정

### Phase 1 (Day 1-2): 정보구조/기본 내비게이션
- 탭 구조 정리(라이브러리/즐겨찾기/설정)
- 라이브러리 홈 기본 레이아웃 구축
- HIG 체크리스트로 헤더/섹션 위계 검증

### Phase 2 (Day 3-5): 프리셋 목록 + 상세 미리보기
- 카드 목록 + 검색 + 카테고리 필터
- 상세 화면 미리보기 영역 구현
- 파라미터 조절 UI 슬라이더/토글 구성

### Phase 3 (Day 6-7): 즐겨찾기
- 로컬 저장(초기: 메모리/로컬 파일)
- 즐겨찾기 탭 목록/빈 상태/삭제 흐름

### Phase 4 (Day 8): 품질 정리
- 라이트/다크 모드 대비 검수
- 탭 타깃/접근성 trait/스냅샷 검수
- 회귀 체크 리스트 실행

### Phase 5 (Day 9-10): 데모 릴리즈 준비
- 데모 시나리오 3개 고정
- 이슈 triage 및 컷라인 확정
- MVP 태그/릴리즈 노트
